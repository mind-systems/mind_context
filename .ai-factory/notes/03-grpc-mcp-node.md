# gRPC from Node.js in stdio MCP Process

**Date:** 2026-03-24
**Source:** research spike (roadmap phase 5.1)
**Full design doc:** mind_mcp/docs/grpc-migration.md

## Key Findings

- Use `@grpc/grpc-js` (pure JS, no native bindings) + `ts-proto` (compile-time stubs) — not runtime proto loading via `@grpc/proto-loader`, which adds cold-start overhead on every process spawn.
- Compile-time stubs generated by `ts-proto` at build time eliminate runtime proto parsing, keep the npm package clean (only `dist/` shipped), and give full TypeScript types.
- PAT auth goes into `grpc.CallCredentials` attached at channel creation via `credentials.combineChannelCredentials` — the token is injected automatically on every RPC with no per-call boilerplate.
- gRPC `ServiceError.code` maps to human-readable strings through a `grpcErrorToMcpError()` utility; existing tool catch blocks need only swap `${err}` for the utility call.

## Details

### Library Stack

`@grpc/grpc-js` as the sole runtime gRPC dependency (pure JS, works in ESM, no native bindings). `ts-proto` as a `devDependency` code generator run at build time (`npm run proto:gen`). `@grpc/proto-loader` is excluded from the runtime bundle entirely.

The MCP process starts fresh per session. Runtime proto loading via `@grpc/proto-loader` parses `.proto` files synchronously on each cold start, adding 50–150ms of file I/O and parsing before any tool call can execute. Compile-time stubs have zero startup cost.

### Proto Loading Strategy

Compile-time only. `.proto` files live in `mind_api/proto/`. `mind_mcp` runs `protoc` + `ts-proto` plugin during `npm run build` (or a separate `npm run proto:gen` step), outputs generated stubs to `src/generated/`, and compiles them with the rest of the TypeScript source. No `.proto` files ship in the npm package.

### PAT Auth via Metadata

```typescript
function makePatCallCredentials(token: string): grpc.CallCredentials {
  return grpc.credentials.createFromMetadataGenerator((_params, callback) => {
    const meta = new grpc.Metadata();
    meta.add("authorization", `Bearer ${token}`);
    callback(null, meta);
  });
}

// At channel creation:
const channelCreds = grpc.credentials.combineChannelCredentials(
  useTls ? grpc.credentials.createSsl() : grpc.credentials.createInsecure(),
  makePatCallCredentials(process.env.MIND_PAT_TOKEN!)
);
const client = new BreathSessionsServiceClient(MIND_GRPC_URL, channelCreds);
```

The `Authorization: Bearer <token>` header from the REST client maps directly to a gRPC metadata key of the same name. No changes to the server-side PAT validation logic are needed.

### Error Mapping

| gRPC status | MCP error string |
|---|---|
| `NOT_FOUND` (5) | Session not found |
| `UNAUTHENTICATED` (16) | Invalid or expired token |
| `PERMISSION_DENIED` (7) | Access denied |
| `INVALID_ARGUMENT` (3) | Invalid request: `<details>` |
| `ALREADY_EXISTS` (6) | Session already exists |
| `RESOURCE_EXHAUSTED` (8) | Rate limit exceeded |
| `UNAVAILABLE` (14) | Service temporarily unavailable — please retry |
| `INTERNAL` (13) | Internal server error |
| `UNIMPLEMENTED` (12) | Operation not supported |
| other | Unexpected error (gRPC code N): `<details>` |

Implemented as `grpcErrorToMcpError(err: ServiceError): string` in `src/api/grpc-errors.ts`.

### grpc-client.ts Sketch

Replaces `src/api/client.ts` with identical exports so all tool files need no changes:

```typescript
// Environment
// MIND_GRPC_URL  — host:port (replaces MIND_API_URL)
// MIND_PAT_TOKEN — unchanged
// MIND_GRPC_TLS  — "true" for TLS, omit for insecure (local dev)

export async function fetchSessions(page?: number, pageSize?: number): Promise<BreathSessionListResponse>
export async function fetchSession(id: string): Promise<BreathSession>
export async function patchSession(id: string, data: Partial<BreathSession>): Promise<BreathSession>
export async function createSession(data: CreateBreathSessionPayload): Promise<BreathSession>
```

Each function wraps the generated callback-based stub in a Promise and converts `ServiceError` via `grpcErrorToMcpError`. A shared `callUnary<Req, Res>()` helper reduces boilerplate. Alternatively, using `ts-proto` with `outputServices=nice-grpc` generates Promise-based stubs natively and is worth evaluating (see Open Questions).

## Open Questions

- **nice-grpc vs raw stubs** — `nice-grpc` package generates cleaner Promise-based stubs with interceptor support; evaluate before writing `grpc-client.ts`.
- **Proto file ownership** — `breath_sessions.proto` must live in `mind_api/` and be referenced by `mind_mcp/` at build time; ownership and path convention not yet decided.
- **SetTimeOfDay RPC shape** — REST `PATCH /breath_sessions/:id` with `{ timeOfDay }` needs a confirmed RPC equivalent (field mask on `UpdateSession` or dedicated `SetTimeOfDay` RPC) before `setTimeOfDay.ts` can be migrated.
- **Self-signed certs in staging** — if staging uses a self-signed cert, `createSsl()` will reject it; `NODE_EXTRA_CA_CERTS` or a bundled CA may be needed.
- **Streaming for classifyAll** — a server-streaming `StreamSessions` RPC would be cleaner than N paginated calls; low priority.
